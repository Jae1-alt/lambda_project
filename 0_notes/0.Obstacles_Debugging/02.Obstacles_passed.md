# ID Token vs Access Token for Testing the API's Cognito Authorizors


# Operational Debugging: ID Token vs. Access Token in API Gateway Cognito Authorizers

## 1. The Issue
When executing a `curl` request to an API Gateway endpoint (`/python`) protected by an Amazon Cognito User Pool Authorizer, the API returned a `{"message":"Unauthorized"}` error. This occurred despite providing a cryptographically valid token generated from a successful Multi-Factor Authentication (MFA) sign-in flow.

The initial request utilizing the Access Token was structured as follows:

```bash
curl https://<api-id>.execute-api.<region>.amazonaws.com/prod/python \
    -H "Authorization: <YOUR_ACCESS_TOKEN_HERE>"
```

## 2. The Solution
The request was modified by swapping the token in the `Authorization` header from the Access Token to the ID Token generated during the `initiate-auth` / `respond-to-auth-challenge` flow.

```bash
curl https://<api-id>.execute-api.<region>.amazonaws.com/prod/python \
    -H "Authorization: <YOUR_ID_TOKEN_HERE>"
```

Substituting the ID Token instantly resolved the unauthorized error, and the API successfully processed the request.

## 3. Architectural Understanding: The Root Cause of the Failure
The `401 Unauthorized` error was not caused by an invalid signature or an expired token. The Access Token was cryptographically valid, but it failed the specific claim structure validation enforced by the API Gateway Cognito Authorizer due to its configuration state.

*   **The Configuration Trigger:** The API Gateway Cognito Authorizer was configured with an empty "Authorization Scopes" list. 
*   **The Default Validation Logic:** When no explicit OAuth scopes are defined on the authorizer, API Gateway defaults to standard identity validation. In this unscoped state, the authorizer strictly requires the presence of the **`aud` (audience)** claim within the JWT, verifying that it matches the configured App Client ID.
*   **The ID Token (Success):** The Cognito ID Token inherently contains the `aud` claim (set to the App Client ID). Therefore, it perfectly satisfies the default validation requirements of the unscoped authorizer.
*   **The Access Token (Failure):** The Cognito Access Token does **not** contain an `aud` claim. Instead, it utilizes a `client_id` claim and a `scope` claim. Because the unscoped authorizer explicitly checks for the `aud` claim, the Access Token fails this specific structural check, resulting in the `401 Unauthorized` response.

## 4. Deep Dive: The `aud` (Audience) Claim in JWTs
To fully understand this failure, it is necessary to examine the mechanics of the `aud` claim, which is the specific point of friction between the Access Token and the API Gateway authorizer.

*   **What it is:** The `aud` (audience) claim is a standard, registered claim defined in the JSON Web Token (JWT) specification (RFC 7519). It identifies the specific recipients or services for which the token is intended.
*   **What it does:** It acts as a critical security boundary to prevent "Token Confusion" or "Replay Attacks." If a user authenticates with Service A, Service A receives a token with `aud: Service_A_ID`. If the user attempts to use that exact same token to access Service B, Service B will check the `aud` claim, see it is intended for Service A, and reject it.
*   **How it works in AWS:** When Amazon Cognito mints an ID Token, it automatically populates the `aud` claim with the specific App Client ID that requested the token. When API Gateway is configured with a Cognito Authorizer, it is explicitly linked to that same App Client ID. During the validation phase, API Gateway extracts the `aud` claim from the incoming JWT and verifies it matches its configured App Client ID. If the claim is missing (as is the case with the Cognito Access Token) or mismatched, the authorizer immediately rejects the token.

## 5. Enabling Access Token Validation
To successfully utilize the Access Token for API authorization, the architecture must be explicitly configured for scope-based validation, shifting the authorizer's focus away from the `aud` claim. This requires:

1. Creating a **Resource Server** within the Amazon Cognito User Pool and defining custom scopes (e.g., `api.read`, `api.write`).
2. Assigning those scopes to the App Client.
3. Populating the **"Authorization Scopes"** field in the API Gateway Authorizer with the exact scope names.

Once scopes are defined in the authorizer, the validation logic shifts from checking the `aud` claim to checking the `scope` claim within the JWT, at which point the Access Token will successfully authenticate the request.
