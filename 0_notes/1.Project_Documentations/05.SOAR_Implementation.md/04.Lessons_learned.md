# Phase 5: Key Lessons Learned & "Gotchas"

Integrating Generative AI into a serverless security pipeline is not just about writing a good prompt; it requires navigating AWS billing mechanics, regional routing constraints, and API fragmentation. Documenting these "gotchas" is critical for maintaining a resilient, enterprise-grade SOAR architecture.

## 1. The AWS Marketplace Billing Gatekeeper (Third-Party vs. First-Party)
* **The Problem:** Despite having valid IAM permissions and enabled Model Access in the console, the Lambda threw an `AccessDeniedException` citing `INVALID_PAYMENT_INSTRUMENT`. 
* **The Lesson:** Amazon Bedrock routes all third-party foundation models (like Anthropic Claude and Meta Llama) through the **AWS Marketplace** backend. AWS enforces a strict gatekeeper rule: an account must have a valid, verified payment instrument (credit/debit card) on file to "subscribe" to these models, even if the account is only using free-tier credits.
* **The Fix / Workaround:** If an account cannot provide a payment method, engineers must pivot to **First-Party AWS Models** (like Amazon Titan Text). Because AWS owns Titan, it completely bypasses the AWS Marketplace EULA and billing gatekeepers, allowing immediate access.

## 2. Cross-Region Inference & IAM Routing
* **The Problem:** Attempting to invoke the base model ID (`anthropic.claude-opus-4-6-v1`) from a Lambda in `us-east-1` resulted in routing errors.
* **The Lesson:** Flagship AI models require massive compute and are not physically hosted in every region. According to the Claude Opus 4.6 Model Card, **In-Region availability is "No" for `us-east-1`**. To use it, you must use the **Geo Cross-Region Inference Profile** (`us.anthropic.claude-opus-4-6-v1`), which routes the request to neighboring regions (like `us-east-2` or `us-west-2`).
* **The Fix:** The IAM policy must explicitly grant `bedrock:InvokeModel` permissions to the `inference-profile/*` ARN, not just the base foundation model ARN. Otherwise, AWS blocks the router from handing the request off to the destination region.

## 3. API Fragmentation: The Multi-Model Payload Trap
* **The Problem:** Assuming that the `bedrock.invoke_model()` API uses a universal JSON schema across all AI providers.
* **The Lesson:** `InvokeModel` is a pass-through API. Every provider dictates their own payload structure. 
    * **Anthropic (Claude):** Requires the `messages` array, the `anthropic_version` string, and parses the response via `content[0].text`.
    * **Amazon (Titan):** Requires the `inputText` string, the `textGenerationConfig` dictionary, and parses the response via `results[0].outputText`.
* **The Fix:** When swapping models to bypass billing gatekeepers or reduce costs, the Python payload builder and the response parser must be entirely rewritten to match the specific provider's schema.

## 4. The Golden Rule: AI Failure Must Not Block Security
* **The Problem:** If the Bedrock API experiences a momentary throttle, a timeout, or a billing error, a standard Python script would crash, leaving stale tokens un-expired in the database.
* **The Lesson:** In a SOAR pipeline, the AI is a luxury (Enrichment), but the workflow is a necessity (Remediation). 
* **The Fix:** The Bedrock API call must be wrapped in a dedicated `try/except` block *inside* the database processing loop. If the AI fails, the exception is caught, a warning is logged, and the script immediately falls through to the `table.update_item()` command. The security remediation always succeeds, regardless of the AI's uptime.

## 5. Separation of Logic and Content (Prompt Management)
* **The Problem:** Hardcoding multi-paragraph SOC prompts inside the Python script makes it impossible for non-technical security analysts to tweak the AI's instructions without a full code deployment.
* **The Lesson:** Application logic (Python) should be strictly separated from content (Prompts).
* **The Fix:** The prompt was extracted into a standalone `soc_prompt_template.txt` file. By updating the Terraform `archive_file` data source to use `source_dir` instead of `source_file`, both the `.py` and `.txt` files are zipped together. The Python script reads the file at runtime using `open()` and injects dynamic variables using `.format()`. This allows prompts to be version-controlled and updated independently of the orchestration logic.
