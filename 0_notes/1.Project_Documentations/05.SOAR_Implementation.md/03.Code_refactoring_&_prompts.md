# Phase 5: Python Refactoring & Prompt Engineering (The Code)

With the infrastructure tuned and the IAM permissions granted, Phase 5 focused on the application layer: integrating the Amazon Bedrock API into the existing Detection Lambda, managing prompts as external configuration, and building a resilient fallback mechanism to ensure security remediation never fails due to AI service disruptions.

## Separation of Logic and Content: The External Prompt Template
In enterprise AI applications, hardcoding multi-paragraph prompts inside Python scripts is considered an anti-pattern. If a security analyst wants to tweak the wording to get better AI results, they should not have to ask a Python developer to change the code, test it, and redeploy the Lambda.

**The Solution:** I extracted the entire SOC analyst prompt into a standalone text file (`soc_prompt_template.txt`) and placed it alongside the Python script. The Python code reads this file into memory at runtime using Python's built-in `open()` function, and injects dynamic variables (like `username` and `timestamp`) using the `.format()` method.

**The Prompt Template (`soc_prompt_template.txt`):**
```text
You are an expert SOC (Security Operations Center) analyst assistant. 

Analyze the following security telemetry event:
- User: {username}
- User Group: {user_group}
- Event Timestamp: {issued_at}
- Event: A JWT token was issued but never used to access protected APIs within the {expiration_minutes}-minute threshold.

Provide your analysis in the following strict format:
1. Severity (Low, Medium, High, Critical)
2. Possible explanations (Bullet points)
3. Recommended analyst actions (Bullet points)
4. Short executive summary (1-2 sentences)
```

This approach enables version control for prompts, allows non-technical security analysts to edit the template, and keeps the Python code focused purely on orchestration.

## Deployment Packaging: `archive_file` and `source_dir`
Because the Lambda now depends on an external `.txt` file, the Terraform deployment pipeline had to be updated. Previously, the `archive_file` data source used `source_file` to zip a single `.py` file. This was changed to `source_dir`, which zips the entire `detection_v3/` folder containing both the Python script and the prompt template.

Crucially, the `code_sha256` argument on the `aws_lambda_function` resource ensures that if even a single character in the `.txt` prompt file changes, Terraform detects the new SHA256 hash and automatically pushes the updated zip to AWS. Without this, Terraform would see that the Python file hasn't changed and skip the deployment entirely.

## The Anthropic Messages API: Payload Structure
According to the AWS Bedrock documentation, Claude Opus 4.6 supports two APIs: `Invoke` and `Converse`. For this lab, I utilized the `Invoke` API via the `bedrock-runtime` endpoint, which passes the request directly through to Anthropic's native Messages API format.

As documented in the [AWS Model Card for Claude Opus 4.6](https://docs.aws.amazon.com/bedrock/latest/userguide/model-card-anthropic-claude-opus-4-6.html), the model is **not hosted In-Region** in `us-east-1`. It is only available via **Geo Cross-Region Inference** (routing to `us-east-1`, `us-east-2`, or `us-west-2`) or **Global Cross-Region Inference**. This means the Model ID passed to the `invoke_model` call must be the Geo Inference ID (`us.anthropic.claude-opus-4-6-v1`), not the base model ID (`anthropic.claude-opus-4-6-v1`).

**The Required JSON Payload:**
```json
{
    "anthropic_version": "bedrock-2023-05-31",
    "max_tokens": 1000,
    "messages": [
        {"role": "user", "content": "The formatted prompt text..."}
    ]
}
```

**Key Anatomy:**
*   **`anthropic_version`**: Must be `"bedrock-2023-05-31"` when calling through AWS Bedrock. This differs from Anthropic's native API outside of AWS.
*   **`max_tokens`**: Mandatory for Claude models. The model supports up to 128K output tokens, but for a concise SOC summary, 1000 is sufficient and cost-effective.
*   **`messages`**: An array that must begin with the `"user"` role. This is the modern Messages API format, replacing the legacy `prompt` string used by older Claude v2 models.

## API Fragmentation: The Multi-Model "Gotcha"
A critical lesson learned during this phase is that **every AI provider uses a different JSON schema** for the `invoke_model` API. If the account's billing gatekeeper blocks third-party models (like Anthropic), and the engineer pivots to a first-party AWS model (like Amazon Titan Text Lite), the entire payload structure must change:

| Feature | Anthropic Claude | Amazon Titan |
| :--- | :--- | :--- |
| **Model ID** | `us.anthropic.claude-opus-4-6-v1` | `amazon.titan-text-lite-v1` |
| **Prompt Key** | `messages` (array) | `inputText` (string) |
| **Config Key** | `max_tokens` | `textGenerationConfig.maxTokenCount` |
| **Response Path** | `content[0].text` | `results[0].outputText` |
| **Billing Gate** | AWS Marketplace (requires payment method) | First-party (no Marketplace EULA) |

This fragmentation reinforces the value of the Terraform `locals` block and environment variables: swapping models requires updating the Model ID and the Python payload parser, but the rest of the orchestration logic remains untouched.

## The Golden Rule Fallback: AI Failure Must Not Block Security
The most critical architectural decision in the Python code is wrapping the Bedrock API call in a dedicated `try/except` block *inside* the token processing loop. 

If Amazon Bedrock experiences a throttle, a timeout, a model unavailability, or a billing error, the inner `except` block catches the exception, logs a warning, and allows the script to continue executing the `table.update_item` command. This ensures that the security remediation (expiring the stale token) **always succeeds**, even if the AI enrichment fails. The AI is a luxury; the security workflow is a necessity.
