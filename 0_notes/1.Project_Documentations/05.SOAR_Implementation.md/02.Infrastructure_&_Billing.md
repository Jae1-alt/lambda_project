# Phase 5: Infrastructure & The Billing Gatekeeper (The Terraform)

Integrating Generative AI into a serverless pipeline requires specific Infrastructure as Code (IaC) adjustments. AI models behave differently than traditional databases or APIs: they are significantly slower to respond, they utilize complex routing profiles, and they are deeply tied to AWS billing mechanisms. This phase focused on tuning the Terraform configuration to support Amazon Bedrock while enforcing the Principle of Least Privilege and DRY (Don't Repeat Yourself) principles.

## DRY Architecture with Terraform `locals`
When configuring AI integrations, the Model ID (e.g., `us.anthropic.claude-opus-4-6-v1`) must be referenced in two distinct places:
1. **The IAM Policy:** To grant the Lambda permission to invoke that specific model.
2. **The Lambda Environment Variables:** To tell the Python `boto3` client which model to call.

Hardcoding the Model ID in both locations creates a "split-brain" risk. If an engineer decides to swap to a cheaper model (like Claude Haiku or Amazon Titan) and forgets to update the IAM policy, the deployment will succeed, but the Lambda will fail at runtime with an `AccessDeniedException`. 

**The Solution:** I utilized a Terraform `locals` block to define the Model ID in a single source of truth. Both the IAM ARN and the Lambda environment variable dynamically interpolate this local variable using `${local.bedrock_model_id}`. This ensures that swapping AI models in the future requires changing exactly one line of code, and the security boundary and application logic will always remain perfectly in sync.

## Tuning Lambda for AI Latency
Calling a DynamoDB table takes milliseconds. Calling a Large Language Model (LLM) to read a prompt, reason through it, and generate a multi-paragraph response takes seconds. 

If a Lambda function is left at the AWS default timeout of 3 seconds, AWS will violently terminate the Python script before Bedrock finishes generating the SOC summary, resulting in a `Task timed out` error. To accommodate the AI's "thinking time," I increased the Lambda `timeout` argument to 60 seconds. Because AWS Lambda bills strictly by the millisecond and stops billing the exact moment the script finishes, raising the timeout ceiling incurs no additional cost for fast responses, but provides the necessary breathing room for the AI to complete its work.

## The IAM "Gotcha": Cross-Region Inference Profiles
Because flagship AI models (like Claude Opus) require massive compute resources, they are not always physically hosted in every single AWS region. To solve this, AWS utilizes **Cross-Region Inference Profiles** (e.g., `us.anthropic.claude-opus-4-6-v1`), which act as intelligent routers that forward requests to available foundation models in neighboring regions.

This introduces a major IAM "gotcha": 
* A standard foundation model ARN looks like: `arn:aws:bedrock:us-east-1:123456789012:foundation-model/anthropic.claude-3-sonnet...`
* A Cross-Region Inference Profile ARN looks like: `arn:aws:bedrock:*:*:inference-profile/us.anthropic...`

If an IAM policy only grants permission to the specific regional foundation model, the Lambda will fail when the router attempts to hand the request off to a different region. The IAM policy must explicitly allow the `bedrock:InvokeModel` action on the `inference-profile/*` ARN to ensure the routing layer functions correctly.

## The AWS Marketplace Billing Gatekeeper (Real-World Gotcha)
During the deployment of this phase, I encountered a critical, real-world roadblock that is rarely discussed in technical tutorials: **The AWS Marketplace Billing Gatekeeper**.

Even if an AWS account has free-tier credits or has not spent any money, Amazon Bedrock routes all third-party foundation models (like Anthropic and Meta) through the AWS Marketplace backend. To prevent abuse, AWS enforces a strict rule: **An account must have a valid, verified payment instrument (credit/debit card) on file** before it will allow the account to "subscribe" to a third-party model.

If the payment method is missing, expired, or fails the initial $1 verification hold, the Bedrock API will reject the `InvokeModel` call with an `INVALID_PAYMENT_INSTRUMENT` or `AccessDeniedException` error, regardless of how perfectly the IAM policies or Python code are written. 

* **The Workaround:** If an account cannot provide a payment instrument, engineers must pivot to **First-Party AWS Models** (like Amazon Titan). Because AWS owns Titan, it bypasses the AWS Marketplace EULA and billing gatekeepers entirely, though it requires modifying the Python payload to match Titan's specific `inputText` JSON schema.
