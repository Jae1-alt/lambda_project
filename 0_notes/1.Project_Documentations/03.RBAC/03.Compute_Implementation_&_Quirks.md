
# RBAC Implementation Module
## Document 3: Compute Implementation & Runtime Quirks

### 3.1 The Compute Layer: My "Defense in Depth" RBAC Logic
While API Gateway handles the initial routing (sending `/python` requests to the Python Lambda, and `/node` requests to the Node Lambda), I realized that relying solely on API Gateway for route enforcement is a security risk. If I accidentally misconfigured the API Gateway routing, a user could execute the wrong business logic.

To mitigate this, I implemented **Defense in Depth** directly inside my Lambda functions. My code explicitly validates two things before executing any logic:
1. **The Route:** It checks the `event.resource` against an expected path.
2. **The Permissions:** It checks the user's `cognito:groups` against an allowed list.

**My Python Implementation**
```python
import json
import os

def lambda_handler(event, context):
    claims = event.get("requestContext", {}).get("authorizer", {}).get("claims", {})
    groups = claims.get("cognito:groups", [])
    resource = event.get("resource")

    expected_resource = os.environ.get("EXPECTED_RESOURCE", "/python")
    allowed_groups = json.loads(os.environ.get("ALLOWED_GROUPS", "[]"))

    # 1. Explicit Route Validation
    if resource != expected_resource:
        return {"statusCode": 404, "body": json.dumps({"error": "Route mismatch."})}

    # 2. Explicit Group Validation (Least Privilege)
    if not any(group in groups for group in allowed_groups):
        return {"statusCode": 403, "body": json.dumps({"error": "Access denied."})}

    return {"statusCode": 200, "body": json.dumps({"message": f"Access granted to {resource}"})}
```

**My Node.js Implementation**
```javascript
exports.handler = async (event) => {
    const claims = event.requestContext?.authorizer?.claims || {};
    const groups = claims["cognito:groups"] || [];
    const resource = event.resource;

    const expectedResource = process.env.EXPECTED_RESOURCE || "/node";
    const allowedGroups = JSON.parse(process.env.ALLOWED_GROUPS || "[]");

    if (resource !== expectedResource) {
        return { statusCode: 404, body: JSON.stringify({ error: "Route mismatch." }) };
    }

    const hasAccess = groups.some(group => allowedGroups.includes(group));
    if (!hasAccess) {
        return { statusCode: 403, body: JSON.stringify({ error: "Access denied." }) };
    }

    return { statusCode: 200, body: JSON.stringify({ message: `Access granted to ${resource}` }) };
};
```

### 3.2 Crossing the Infrastructure Boundary: Environment Variable Data Types
To make these Lambda functions dynamic and reusable, I passed the `expected_resource` and `allowed_groups` via AWS Lambda Environment Variables. This kept my infrastructure configuration (Terraform) completely decoupled from my application code.

However, I quickly ran into a critical architectural constraint: **AWS Lambda environment variables can only hold strings.** 

This created a data-type mismatch when passing lists from Terraform to the compute layer. To resolve this, I had to serialize the data in Terraform and deserialize it in my Lambda code.

**1. Terraform (Serialization)**
When defining the `env_value` map, I passed simple strings directly. However, lists had to be wrapped in the `jsonencode()` function to convert them into a JSON-formatted string.
```hcl
env_value = {
  EXPECTED_RESOURCE = "/python"               # Passed as a raw string
  ALLOWED_GROUPS    = jsonencode(["admin", "student"]) # Passed as the string '["admin","student"]'
}
```
*My Crucial Gotcha:* I learned that if I wrapped a simple string in `jsonencode()` (e.g., `jsonencode("/python")`), Terraform would output the literal string `""/python""` (with double quotes included). This caused silent string-matching failures in my Python code. I established a strict rule for myself: only use `jsonencode()` for lists and maps, never for raw strings.

**2. Compute Layer (Deserialization)**
When my Lambda functions read the environment variable, they received a string. To use the list in my logic, I had to parse the string back into a native list/array.
*   **In Python:** I used `json.loads()` to convert the JSON string back into a Python list.
*   **In Node.js:** I used `JSON.parse()` to convert the JSON string back into a JavaScript Array.

### 3.3 Architectural Debugging: Runtime and Import Quirks
During the deployment and testing of these functions, I encountered two specific runtime errors that highlighted the strict differences between the Python and Node.js execution environments.

**Debugging Lesson 1: The Lambda Handler Dot Notation Crash**
*   **The Error:** `Runtime.ImportModuleError: Unable to import module 'python_rbac_v1.5': No module named 'python_rbac_v1'`
*   **The Cause:** I had named my file `python_rbac_v1.5.py` and set the Terraform handler to `python_rbac_v1.5.lambda_handler`. I learned that in Python, the dot (`.`) is strictly reserved as a separator for packages and modules. The Lambda runtime interpreted `python_rbac_v1` as a directory and `5` as the file inside it. 
*   **The Fix:** Because Python module names cannot contain dots, I renamed my file to `python_rbac_v1_5.py` and updated my Terraform handler to `python_rbac_v1_5.lambda_handler`.

**Debugging Lesson 2: The Missing Standard Library Import**
*   **The Error:** `NameError: name 'os' is not defined`
*   **The Cause:** My Python script attempted to use `os.environ.get()` without importing the `os` module first. Unlike some languages that load all standard libraries by default, Python requires explicit imports to keep the execution environment lightweight.
*   **The Fix:** I added `import os` and `import json` to the top of my Python file. 
*   **The Node.js Contrast:** I noted that this error does not occur in my Node.js function because environment variables and JSON parsing are handled by global objects (`process.env` and `JSON.parse()`). These global objects are automatically available in the Node.js runtime without requiring any `require()` or `import` statements.
