
# RBAC Implementation Module

## Document 2: Infrastructure Implementation & Terraform Quirks

### 2.1 Separation of Concerns: Roles vs. Assignments
To maintain a clean, scalable, and readable Infrastructure as Code (IaC) architecture, I made the deliberate decision to strictly separate the definition of my roles (Cognito Groups) from the assignment of users to those roles. This separation prevented circular dependencies and mirrored real-world Identity and Access Management (IAM) practices.

**Defining the Groups**
I defined my groups as a simple list of strings. Because the `for_each` meta-argument requires a set or a map, I utilized the `toset()` function to convert the list into a set of unique strings.

```hcl
variable "cognito_groups" {
  type    = list(string)
  default = ["admin", "student"]
}

resource "aws_cognito_user_group" "this" {
  for_each     = toset(var.cognito_groups)
  name         = each.value
  user_pool_id = aws_cognito_user_pool.main.id
}
```

**Defining the Users and their Roles**
I defined my users in a map of objects. Crucially, I set the `groups` attribute as a `list(string)`. This allowed a single user to be assigned to multiple roles, establishing a "many-to-many" relationship.

```hcl
variable "lambda_cognito_users" {
  type = map(object({
    username = string
    email    = string
    groups   = list(string)
  }))
  
  default = {
    admin_user = {
      username = "admin1"
      email    = "admin1@example.com"
      groups   = ["admin", "student"] 
    }
    student_user = {
      username = "student1"
      email    = "student1@example.com"
      groups   = ["student"]
    }
  }
}
```

### 2.2 My Terraform "Flattening" Engine: Solving the Many-to-Many Problem
To create the `aws_cognito_user_in_group` resources, I faced a structural challenge: the `for_each` meta-argument requires a flat, one-dimensional map, but my variable was a nested structure (a map of users, containing lists of groups). Terraform cannot natively loop through a list inside a map to create a resource.

To solve this, I built a Terraform `locals` block to act as a "flattening engine." I used a nested `for` expression combined with the `flatten()` function to transform my hierarchical data into a flat map of unique user-group pairs.

```hcl
locals {
  # 1. I iterate through users, then through their specific groups, creating a list of objects
  user_group_pairs = flatten([
    for user_key, user_val in var.lambda_cognito_users : [
      for group in user_val.groups : {
        unique_key = "${user_key}-${group}"
        username   = user_key
        group      = group
      }
    ]
  ])

  # 2. I convert the flat list of objects into a map keyed by the unique_key
  user_group_map = {
    for pair in local.user_group_pairs : pair.unique_key => pair
  }
}
```

**How My Engine Works (Step-by-Step):**
1. The **outer loop** iterates through my `lambda_cognito_users` map.
2. The **inner loop** iterates through the `groups` list *inside* that specific user's object.
3. This generates a "list of lists" containing objects for every valid combination (e.g., `admin_user-admin`, `admin_user-student`, `student_user-student`).
4. The **`flatten()`** function crushes those nested lists into a single, one-dimensional list.
5. The final **`for` expression** converts that list into a map, using the `unique_key` (e.g., `"admin_user-admin"`) as the map key, which `for_each` strictly requires.

With this flattened map, creating the associations became incredibly clean and dynamic:

```hcl
resource "aws_cognito_user_in_group" "this" {
  for_each     = local.user_group_map
  user_pool_id = aws_cognito_user_pool.main.id
  group_name   = each.value.group
  username     = each.value.username
}
```

### 2.3 Architectural Debugging: Terraform Type Unification in Default Blocks
While building the variables for my Lambda environment configurations, I encountered a notorious Terraform "gotcha" regarding how it handles mixed data types inside `default` blocks.

**The Scenario:**
I wanted to pass both simple strings (like the API route `"/python"`) and lists (like the allowed groups `["admin", "student"]`) into a single `env_value` map inside a variable's `default` block.

**The Error I Hit:**
```text
Error: Invalid default value for variable
This default value is not compatible with the variable's type constraint: cannot find a common base type for all elements.
```

**The "Why":**
I learned that even when you explicitly tell a variable to accept `any` type, Terraform's HCL parser attempts to unify the types of the *literal values* inside a `default = { ... }` block before it applies the type constraint. It looked at the string `"/python"`, assumed the map was a `map(string)`, hit the list `["admin", "student"]`, failed to find a strict common base type for the literal elements, and crashed.

**How I Solved It:**
I used the pragmatic approach (Structural Parity). I ensured every object in my map had the exact same structure. If my Node function didn't need groups, I still provided an empty list `[]` so Terraform saw a list in both objects, satisfying the type unifier.

```hcl
# Inside my default block
python = { env_value = { ALLOWED_GROUPS = ["admin"] } }
node   = { env_value = { ALLOWED_GROUPS = [] } } # Satisfies the type checker
```

By understanding this quirk, I avoided hours of frustration and established a clean pattern for handling complex, nested data structures in my Terraform code.
