# WAF Telemetry & AI Analysis: Infrastructure & The CloudWatch Race Condition

Integrating AWS WAF with serverless compute requires navigating strict AWS naming conventions, resource lifecycle behaviors, and IaC syntax traps. This phase focused on writing resilient Terraform configurations that prevent state drift, optimize security testing, and ensure log telemetry flows correctly from the edge to the database.

## 1. WAFv2 Rate-Based Rules & Custom Evaluation Windows
Historically, AWS WAF Rate-Based rules operated on a fixed 5-minute sliding window with a hard minimum threshold of 100 requests. This made testing in lab environments tedious, as engineers had to write scripts to blast their APIs with hundreds of requests just to trigger a single block.

With WAFv2, AWS introduced **Custom Evaluation Windows** via the `evaluation_window_sec` argument. 
* **The Configuration:** By setting `limit = 10` and `evaluation_window_sec = 300`, the WAF evaluates traffic over a 5-minute window but drops the minimum threshold to just 10 requests.
* **The Engineering Value:** This creates a highly realistic testing environment for credential-stuffing or brute-force defenses. Engineers can trigger a `403 Forbidden` block with a simple 15-iteration bash loop, allowing for rapid iteration and verification of the SOAR pipeline without generating massive artificial traffic spikes.

```hcl
rate_based_statement {
  limit                 = 10
  aggregate_key_type    = "IP"
  evaluation_window_sec = 300 # Custom 5-minute window
}
```

## 2. The CloudWatch Log Group Race Condition
One of the most common "gotchas" in enterprise Lambda deployments is the CloudWatch Log Group Race Condition. 

By default, if a Lambda function executes and its designated CloudWatch Log Group does not exist, the Lambda service will **auto-create the log group on the fly**. However, when AWS auto-creates a log group, it sets the retention policy to **"Never Expire"**. 

If Terraform later attempts to manage that log group and apply a strict 14-day retention policy (to save on storage costs and maintain compliance), it will encounter state drift or fail to apply the retention update.

**The Senior Engineer Fix:** 
Explicitly define the CloudWatch Log Groups in Terraform *before* the Lambda function is created, and use the `depends_on` meta-argument to enforce the creation order.

```hcl
# 1. Explicitly create the log group with a strict retention policy
resource "aws_cloudwatch_log_group" "waf_analyzer_logs" {
  name              = "/aws/lambda/waf-bedrock-analyzer"
  retention_in_days = 14
}

# 2. Force Terraform to create the log group BEFORE the Lambda
resource "aws_lambda_function" "waf_analyzer" {
  function_name = "waf-bedrock-analyzer"
  # ... other config ...
  
  depends_on = [
    aws_cloudwatch_log_group.waf_analyzer_logs
  ]
}
```

## 3. The WAF Log Group Naming Strictness & Association
When configuring WAF to stream telemetry to CloudWatch (via `aws_wafv2_web_acl_logging_configuration`), AWS enforces a notoriously strict, undocumented naming convention.

The CloudWatch Log Group name **must** begin with the prefix `aws-waf-logs-`. 
* **Valid:** `aws-waf-logs-api-gateway-prod`
* **Invalid:** `/aws/waf/my-api-logs`

If the log group is named incorrectly, AWS WAF will silently refuse to stream the logs. The WAF Web ACL will appear healthy in the console, but the CloudWatch Log Group will remain empty, leading to massive troubleshooting headaches when the Detective Lambda attempts to query the logs and finds nothing.

Furthermore, simply creating a WAF WebACL is not enough. You must explicitly bind it to the API Gateway Stage using the `aws_wafv2_web_acl_association` resource. Because WAF operates at the absolute edge, this association is what allows WAF to count and block requests (returning `403 Forbidden`) *before* API Gateway ever evaluates internal method authorization (which would otherwise return `401 Unauthorized`).
