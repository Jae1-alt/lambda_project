# WAF Telemetry & AI Analysis: Advanced IAM & The `templatefile` Bridge

In enterprise environments, Identity and Access Management (IAM) for serverless applications is rarely static. Hardcoding AWS Account IDs, Regions, or Resource ARNs into IAM policies creates fragile code that breaks when moving from Development to Production. This phase focused on bridging the gap between static JSON policy files and dynamic Terraform infrastructure, while strictly enforcing the Principle of Least Privilege.

## 1. Separation of Concerns: Execution vs. Business Logic
A common anti-pattern in serverless IAM is granting a Lambda function blanket `AdministratorAccess` or overly broad `*` permissions to "make the errors go away." A Senior Engineer approach splits IAM into two distinct layers:

*   **The Baseline (AWS Managed Policy):** The `AWSLambdaBasicExecutionRole` is attached to the Lambda's execution role. This grants the fundamental permissions required for the Lambda service to boot up and write its own execution logs (`logs:CreateLogStream`, `logs:PutLogEvents`) to its specific `/aws/lambda/...` log group.
*   **The Business Logic (Custom Inline Policy):** A separate, strictly scoped inline policy is created *only* for the application's specific tasks. This policy grants:
    *   `logs:FilterLogEvents` (Scoped specifically to the WAF telemetry log group).
    *   `dynamodb:PutItem` (Scoped specifically to the `waf-events` table).
    *   `bedrock:InvokeModel` (Scoped to the AI model).

This separation ensures that if the custom policy is ever audited or modified, the baseline logging capabilities of the Lambda remain untouched.

## 2. Bridging Static JSON and Dynamic Terraform
Security and compliance teams often prefer to review IAM policies in standard `.json` files rather than reading Terraform HCL (`jsonencode`) blocks. To support this, Terraform provides the `templatefile()` function, which allows engineers to inject dynamic variables into static text files.

**The JSON Template (`cw_waf_log_access.json`):**
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadWAFLogs",
      "Effect": "Allow",
      "Action": ["logs:FilterLogEvents"],
      "Resource": "${log_group_arn}:*"
    },
    {
      "Sid": "WriteToDynamoDB",
      "Effect": "Allow",
      "Action": ["dynamodb:PutItem"],
      "Resource": "${dynamodb_table_arn}"
    }
  ]
}
```

**The Terraform Implementation:**
```hcl
resource "aws_iam_role_policy" "waf_analyzer_custom_policy" {
  name = "waf-analyzer-business-logic"
  role = aws_iam_role.waf_lambda_role.id

  policy = templatefile("${path.module}/policies/cw_waf_log_access.json", {
    log_group_arn      = aws_cloudwatch_log_group.waf_logs.arn
    dynamodb_table_arn = aws_dynamodb_table.waf_events.arn
  })
}
```
When `terraform apply` runs, Terraform reads the JSON file, replaces `${log_group_arn}` with the actual ARN generated during the deployment, and pushes the perfectly scoped policy to AWS.

## 3. The Tuple Trap: Debugging `for_each` Interpolation
When utilizing Terraform loops (`for_each` or `count`) to create multiple resources (e.g., creating Log Groups for multiple Lambdas simultaneously), referencing the resource's ARN in a `templatefile` block will trigger a critical error:

`Call to function "templatefile" failed: Invalid template interpolation value; Cannot include the given value in a string template: string required, but have tuple.`

**The Root Cause:** 
Because `for_each` creates a *map* of resources, asking Terraform for `aws_cloudwatch_log_group.my_logs.arn` doesn't return a single string; it returns a tuple (a list) of *all* the ARNs created in the loop. The `templatefile` function panics because it cannot shove a list into a single string placeholder.

**The Fix (Bracket Notation):**
You must explicitly tell Terraform which key from the loop you want to target using bracket notation `["key"]`:
```hcl
  policy = templatefile("${path.module}/policies/cw_waf_log_access.json", {
    # Target the specific "waf" key from the for_each loop
    log_group_arn = aws_cloudwatch_log_group.my_logs["waf"].arn 
  })
```

## 4. The JSON Syntax Traps (Learning from Raw Repositories)
When translating raw IAM policies from documentation, markdown files, or GitHub repositories into Terraform, engineers frequently encounter parsing failures. Raw snippets often contain "helpful" annotations that break strict JSON parsers. 

Based on standard WAF lab repositories, here are the three most common traps:

### Trap A: Comments are Invalid
JSON does not support `#` or `//` comments. Including lines like `# later use "Resource": "arn:aws:logs..."` will cause Terraform's `templatefile` to crash, often throwing a confusing `Invalid multi-line string` error because it misinterprets the comment syntax as unclosed quotation marks.
* **The Fix:** Always strip all comments from `.json` policy files before passing them to Terraform.

### Trap B: Missing Commas in Arrays
In IAM action arrays, it is incredibly common to miss a comma between actions.
```json
"Action": [
  "dynamodb:PutItem"     <--- MISSING COMMA HERE
  "dynamodb:GetItem",
  "dynamodb:Query"
]
```
* **The Fix:** Validate JSON syntax locally using the terminal command `python3 -m json.tool policy.json` before running `terraform apply`.

### Trap C: Hardcoded Placeholders vs. Dynamic Interpolation
Raw snippets often use placeholders like `"arn:aws:dynamodb:<region>:<account-id>:table/waf-events"`. If copied directly, AWS will literally see the text `<account-id>` and deny access.
* **The Fix:** Replace all hardcoded placeholders with Terraform template variables (e.g., `${dynamodb_table_arn}`) so the IaC engine can inject the exact, correct ARN at runtime.
