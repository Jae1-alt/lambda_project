# Document 4: Key Lessons Learned & "Gotchas"

Building this stateful, serverless architecture required navigating several strict AWS and Terraform constraints. Documenting these "gotchas" is critical for my future self and for anyone maintaining or scaling this system. These are the hard-won battles that transformed this project into a robust enterprise solution.

## 1. The DynamoDB "Boolean Trap" in GSIs
* **The Problem:** Initially, my telemetry scripts wrote the `used` attribute as a native Python Boolean (`True`/`False`). However, when attempting to use this attribute as a Key in a Global Secondary Index (GSI), the system failed.
* **The Lesson:** DynamoDB strictly forbids native Booleans from being used as Partition or Sort keys in GSIs. GSI keys must be of type String, Number, or Binary.
* **The Fix:** I refactored the entire data model to use Strings (`"true"`, `"false"`, `"expired"`). This not only satisfied the GSI schema requirements but also enabled a third state (`"expired"`), which was crucial for the auditability features in Phase 4.

## 2. IAM Permissions Do Not Cascade to Indexes
* **The Problem:** My Detection Lambda was granted `dynamodb:Query` and `dynamodb:UpdateItem` permissions on the base table's ARN, but it still threw an `AccessDeniedException` when trying to query the GSI.
* **The Lesson:** In AWS IAM, a Global Secondary Index is treated as a completely separate physical resource. Permissions granted to the base table do not automatically cascade to its indexes.
* **The Fix:** I had to update the IAM policy's `Resource` block to an array that explicitly includes both the base table and a wildcard for all its indexes: `["${dynamodb_table_arn}", "${dynamodb_table_arn}/index/*"]`.

## 3. Terraform's "Unused Attributes" Enforcement
* **The Problem:** Defining an attribute (like `username`) in the top-level `attribute {}` block of the Terraform DynamoDB resource, but not using it as a key in the base table or a GSI, resulted in a deployment failure (`Error: all attributes must be indexed. Unused attributes...`).
* **The Lesson:** Terraform enforces a strict boundary between the database schema (Keys) and the payload (Data). If you declare an attribute in Terraform, you are explicitly telling Terraform you intend to use it for indexing.
* **The Fix:** Attributes that are only used as payload data should be completely omitted from the Terraform `attribute {}` blocks. The Python scripts can still write and read this data dynamically, but Terraform remains blissfully unaware of it, keeping the infrastructure code clean and error-free.

## 4. The "Query" vs. "Scan" Access Pattern Mismatch
* **The Problem:** The Detection Lambda initially attempted to use a `Query` operation on a GSI to find all unused tokens, but DynamoDB threw a `ValidationException` demanding an equality condition for the HASH key.
* **The Lesson:** DynamoDB's `Query` operation is incredibly fast, but it is strictly bound by the rule: *You must provide the Partition Key (HASH).* You cannot use a `Query` to search across all partitions. If your access pattern requires searching across the whole table without knowing the HASH key, you must either build a new GSI where the desired search attribute is the HASH key, or fall back to a `Scan`.
* **The Fix:** This realization drove the architectural decision to build a dedicated GSI specifically for the Janitor's access pattern (HASH: `used`, RANGE: `issued_at`), ensuring the maintenance pipeline remained highly efficient and cost-effective.

## 5. Lambda Execution Context & Connection Reuse
* **The Problem:** Initial Lambda executions were experiencing unnecessary latency.
* **The Lesson:** AWS Lambda "freezes" the execution environment after a request and "thaws" it for the next one. If the AWS SDK client (e.g., `boto3.resource`) is initialized *inside* the handler function, the Lambda is forced to establish a brand new TCP connection to DynamoDB on every single invocation.
* **The Fix:** I moved the SDK initialization *outside* the handler function. This allows "warm" Lambda invocations to reuse the existing, cached TCP connection, drastically reducing latency and improving API responsiveness.

## 6. Terraform Type Systems: Sets vs. Lists (To be updated)
* **The Problem:** When dynamically generating the Cognito Hosted UI login URL in the Terraform outputs, attempting to grab the first callback URL using `callback_urls[0]` threw an error: *"Elements of a set are identified only by their value and don't have any separate index..."*
* **The Lesson:** The AWS provider returns certain attributes (like Cognito callback URLs and OAuth scopes) as a `set` (an unordered collection), not a `list` (an ordered collection). Terraform strictly forbids indexing a `set` with `[0]`.
* **The Fix:** I used the `tolist()` function to cast the set into a list before indexing it, ensuring the Terraform output generated the correct redirect URI dynamically without throwing type errors.

* **IAM Policy Elements for DynamoDB:**
  * [AWS Documentation: Specifying Resources in IAM Policies](https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_elements_resource.html) - Explains the ARN formats for DynamoDB, specifically the `/index/*` suffix required for GSI permissions.
* **Terraform DynamoDB Resource:**
  * [Terraform Registry: aws_dynamodb_table](https://registry.terraform.io/providers/hashicorp/aws/latest/docs/resources/dynamodb_table) - Documents the `attribute` block and the strict requirement that all defined attributes must be used in a `key_schema`.
* **DynamoDB Query vs. Scan:**
  * [AWS Documentation: Query and Scan](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Query.html) - The definitive guide on the differences, performance implications, and key requirements for `Query` vs `Scan`.
* **Lambda Best Practices:**
  * [AWS Documentation: Best Practices for Working with AWS SDKs](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html#best-practices-sdk) - Officially recommends initializing SDK clients outside the handler for connection reuse.
* **Terraform Type Conversion Functions:**
  * [Terraform Language Documentation: tolist](https://developer.hashicorp.com/terraform/language/functions/tolist) - The official reference for converting sets to lists to enable indexing.
